Configuration Management of microservices using Consul Distributed KV Store:

Overview:

This project was done when I was an upstream contributor to Open Network Automation Platform project.

The main purpose of this system is to provide a scalable central configuration management to manage various configuration files of various microservices and to push changes them without having the microservices to undergo restart thereby helping them scale well.

I am currently maintaining the project at: https://git.onap.org/music/distributed-kv-store/tree/

Languages and Tools:

  • Go (Golang) 1.10
  • Gorilla Mux
  • Hashicorp Consul Distributed Key Value store
  • Docker
  • Kubernetes Helm

Working:

  • The admin of each microservice registers a microservice with the service. They get back a unique token which is used along with a key.
  • Using the token, the admin will upload a configuration files. The configuration files typically have key values such as "DP_IP=IP address" or "ImagePullPolicy=Always" etc.
  • On the microservices side, the client code will keep looking at Consul for the "token+key" for key changes. This is possible using Consul supported client such as Config4j or Python Consul client.
  • When the config as loaded into Consul, those values are now dynamically pushed to the microservices without having them to undergo restarts as the microservices are constantly looking for changes.
  • The whole application is packaged as Kubernetes Helm Charts to be deployed on a Kubernetes Cluster.
